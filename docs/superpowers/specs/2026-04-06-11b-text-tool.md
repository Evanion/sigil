# Spec 11b — Text Tool

## Overview

Adds text node creation, inline canvas editing, and typography controls. Text is rendered on canvas for display and uses an HTML overlay for editing (native cursor, selection, IME input). Plain text (single style per node) for v1, with a data model that supports future rich text (per-range styles).

**Depends on:** Spec 01 (core engine — TextContent type exists), Spec 04 (frontend editor), Spec 11a (select tool enhancements)

---

## 1. Text Node Data Model

### 1.1 Core Type Extension

The core `TextContent` type is extended to support typography:

```rust
pub struct TextStyle {
    pub font_family: String,       // e.g., "Inter", "system-ui"
    pub font_size: f64,            // in px, must be > 0 and finite
    pub font_weight: u16,          // 100–900 (CSS font-weight scale)
    pub font_style: FontStyle,     // normal | italic
    pub line_height: LineHeight,   // auto | fixed(f64)
    pub letter_spacing: f64,       // in px, can be negative, must be finite
    pub text_align: TextAlign,     // left | center | right | justify
    pub text_decoration: TextDecoration, // none | underline | strikethrough
    pub color: StyleValue<Color>,  // text color (token-ref compatible)
}

pub enum FontStyle { Normal, Italic }
pub enum LineHeight { Auto, Fixed(f64) }
pub enum TextAlign { Left, Center, Right, Justify }
pub enum TextDecoration { None, Underline, Strikethrough }

pub enum TextSizing {
    AutoWidth,     // text box grows horizontally, no wrapping
    FixedWidth,    // text wraps within the node's width
    AutoHeight,    // (future) height auto-adjusts to content
}
```

### 1.2 Text Node Kind

```rust
NodeKind::Text {
    content: String,          // plain text content (UTF-8)
    style: TextStyle,         // single style for v1
    sizing: TextSizing,       // auto-width or fixed-width
    // Future: spans: Vec<TextSpan> for rich text
}
```

### 1.3 Future Rich Text Model (Not Implemented)

Documented for forward compatibility. The schema allows adding a `spans` field later:

```rust
// Future — not part of v1
pub struct TextSpan {
    pub start: usize,       // byte offset into content
    pub end: usize,         // byte offset (exclusive)
    pub style: TextStyle,   // override style for this range
}
```

When `spans` is empty (v1), the node-level `style` applies to all text. When spans are present (future), each span overrides the base style for its character range.

## 2. Text Tool Interaction

### 2.1 Tool Activation

The text tool is activated by pressing **T** or clicking the text tool in the toolbar. Cursor changes to a text cursor (crosshair with "T" or standard text-creation cursor).

### 2.2 Creating Text Nodes

**Click (no drag):** Creates an auto-width text node at the click position. Default size: font-size determines initial height, width expands as user types. Immediately enters edit mode.

**Click+drag:** Creates a fixed-width text node with the dragged bounding box. Text wraps within the width. Height is determined by content. Immediately enters edit mode.

**Minimum dimension:** Same as shape tool — if drag dimensions are below 1px, treat as a click (auto-width).

**Default text style:**
- Font: "Inter" (project default, falls back to system-ui)
- Size: 16px
- Weight: 400 (regular)
- Style: normal
- Line height: auto (1.5× font size)
- Letter spacing: 0
- Alignment: left
- Decoration: none
- Color: black `{ space: "srgb", r: 0, g: 0, b: 0, a: 1 }`

### 2.3 Editing Text

**Enter edit mode:** Double-click an existing text node, or immediately after creating one.

**Edit mode behavior:**
1. An HTML `contenteditable` div is positioned exactly over the text node in screen coordinates
2. The div inherits the text node's font properties via CSS
3. The canvas text is hidden for the edited node (the overlay replaces it)
4. The user types, selects, copies, pastes using native browser behavior
5. On each input event, the text content is synced to the store (optimistic update)

**Exit edit mode:**
- Click outside the text node → commit and deselect
- Press Escape → commit and return to select tool, node stays selected
- Press Enter on auto-width text → insert newline (Shift+Enter not needed)
- Select another tool → commit and switch

**Commit:** On exit, send a `SetTextContent` mutation with the final text string. If text is empty, delete the node (empty text nodes are not useful).

### 2.4 Text Tool State Machine

```
idle (text tool active)
  → click on empty canvas → create auto-width node, enter editing
  → click+drag on empty canvas → create fixed-width node, enter editing
  → click on existing text node → select it
  → double-click on existing text node → enter editing

editing
  → input events → sync content to store
  → click outside → commit, exit to idle
  → escape → commit, switch to select tool
  → tool change → commit, switch tool
```

## 3. Canvas Text Rendering

### 3.1 Display Rendering

When NOT in edit mode, text is rendered on canvas using `ctx.fillText()`.

**Font string:** Constructed from TextStyle fields: `"${fontStyle} ${fontWeight} ${fontSize}px ${fontFamily}"`

**Line wrapping (fixed-width):** Words are split by whitespace. Each word is measured with `ctx.measureText()`. Words are accumulated on a line until the next word would exceed the node width. A new line is started. Lines are rendered top-to-bottom with the computed line height.

**Auto-width:** No wrapping. Newlines (`\n`) create line breaks. Node width is the maximum line width (measured). Node height is total lines × line height.

**Text alignment:** Within each line, offset the x position:
- Left: x = 0
- Center: x = (nodeWidth − lineWidth) / 2
- Right: x = nodeWidth − lineWidth
- Justify: distribute extra space between words (skip for last line)

**Vertical positioning:** First baseline at `y + fontSize` (approximate ascent). Subsequent lines at `y + lineHeight * lineIndex`.

### 3.2 Text Measurement

A `text-measure.ts` module provides:

```typescript
measureTextNode(
  content: string,
  style: TextStyle,
  sizing: TextSizing,
  maxWidth?: number  // for fixed-width
): { width: number; height: number; lines: MeasuredLine[] }

interface MeasuredLine {
  text: string;
  width: number;
  y: number;  // baseline y offset from top
}
```

Used by both the renderer (for drawing) and the resize logic (auto-width nodes auto-size to content).

### 3.3 Font Loading

For v1, use system fonts and a bundled web font (Inter). Font loading uses the CSS Font Loading API (`document.fonts.load()`).

**Font stack:** `"Inter", "system-ui", "-apple-system", "Segoe UI", "sans-serif"`

**Future:** A font picker panel with Google Fonts or custom font upload. Not in scope for v1.

## 4. HTML Editing Overlay

### 4.1 Overlay Positioning

A `text-overlay.ts` module manages the editing overlay:

```typescript
createTextOverlay(
  node: DocumentNode,
  viewport: Viewport,
  canvasElement: HTMLCanvasElement
): HTMLDivElement
```

The overlay div is:
- Positioned absolutely over the canvas element
- Transformed to match the node's screen-space position and size (using viewport conversion)
- Styled to match the text node's font properties exactly
- Given `contenteditable="true"` and `spellcheck="false"`
- Focused immediately after creation

### 4.2 Viewport Sync

When the user pans or zooms while editing, the overlay must reposition to stay aligned with the canvas node. A `createEffect` watches the viewport signal and updates the overlay's CSS transform.

### 4.3 Content Sync

On each `input` event in the overlay div:
1. Read `textContent` from the div (plain text, preserving newlines)
2. Update the store's text content optimistically
3. The canvas re-renders other nodes but skips the edited node (overlay is visible instead)

On commit (exit edit mode):
1. Read final `textContent`
2. Send `SetTextContent` mutation to server
3. Remove overlay div from DOM
4. Re-render the node on canvas

### 4.4 Accessibility

The overlay div gets:
- `role="textbox"`
- `aria-label="Edit text"` (or the node name)
- `aria-multiline="true"`
- Native keyboard navigation (arrow keys, Home/End, Cmd+A)
- Native clipboard (Cmd+C/V/X)

Since the overlay is a real HTML element, screen readers can interact with it natively during edit mode.

## 5. Typography Controls

### 5.1 Properties Panel Integration

When a text node is selected, the Layout tab in the Design panel shows typography controls:

**Typography section** (below position/size):
- Font family — text input with dropdown (system fonts for v1)
- Font size — NumberInput with px suffix
- Font weight — Select dropdown (Thin 100 → Black 900)
- Font style — toggle button (I for italic)
- Line height — NumberInput (auto toggle + px value)
- Letter spacing — NumberInput with px suffix
- Text align — 4-button segmented control (left, center, right, justify)
- Text decoration — toggle buttons (underline, strikethrough)
- Text color — ColorSwatch (reuses existing component)

### 5.2 Store Mutations

New store methods:

```typescript
setTextContent(uuid: string, content: string): void
setTextStyle(uuid: string, style: Partial<TextStyle>): void
```

`setTextStyle` accepts a partial — only changed fields are merged with the existing style. This avoids sending the entire style on every keystroke in a NumberInput.

### 5.3 Core Commands

**`SetTextContent` command** — already exists in Spec 01. Validates content length (MAX_TEXT_CONTENT_LENGTH).

**`SetTextStyle` command** — new. Validates:
- font_size > 0 and finite
- font_weight in 100–900 range
- letter_spacing finite
- line_height (if fixed) > 0 and finite

Single undo step. Captures previous style for rollback.

## 6. Keyboard Shortcuts

| Shortcut | Action |
|----------|--------|
| T | Activate text tool |
| Double-click text node | Enter edit mode |
| Escape (in edit mode) | Commit and exit to select tool |
| Ctrl/Cmd + B (in edit mode) | Toggle bold (400↔700) for v1 (future: per-selection) |
| Ctrl/Cmd + I (in edit mode) | Toggle italic for v1 (future: per-selection) |
| Ctrl/Cmd + U (in edit mode) | Toggle underline for v1 (future: per-selection) |

Note: In v1, bold/italic/underline toggles apply to the entire text node (plain text model). In the future rich text model, they would apply to the current selection within the text.

## 8. Input Validation

- **Text content:** Max length `MAX_TEXT_CONTENT_LENGTH` (defined in core `validate.rs`). UTF-8 string.
- **Font size:** Must be > 0, finite. Enforced by `SetTextStyle` command.
- **Font weight:** Must be 100–900 in increments of 100. Enforced by command.
- **Letter spacing:** Must be finite (can be negative). Enforced by command.
- **Line height (fixed):** Must be > 0, finite. Enforced by command.
- **Font family:** Non-empty string, max length `MAX_NAME_LENGTH`. No path characters.
- **All NumberInput values:** Guarded with `Number.isFinite()` at the UI boundary per CLAUDE.md §11.

---

## 9. Consistency Guarantees

- **SetTextStyle atomicity:** All style fields update together. Undo restores the complete previous style.
- **SetTextContent atomicity:** Content replacement is all-or-nothing. Undo restores previous content.
- **Edit overlay sync:** Content is synced optimistically on input events but only committed to the server on exit. If the overlay is abandoned (e.g., page navigation), the last synced content is preserved.
- **Auto-size consistency:** When text content changes, the node's width/height is recomputed from the text measurement. The transform update is part of the same mutation.

---

## 10. WASM Compatibility

No new external dependencies in the core crate. `TextStyle`, `FontStyle`, `TextAlign`, `TextDecoration`, `LineHeight`, `TextSizing` are all plain enums and structs. No I/O, no system calls, no `Send`/`Sync` bounds.

Text measurement happens in the frontend (browser's `measureText` API), not in the core crate. The core only stores the text content and style — it does not perform layout.

---

## 11. Recursion Safety

No recursive algorithms. Text wrapping is an iterative word-by-word scan. Line layout is a flat loop.

## 12. PDR Traceability

**Implements:**
- PDR §4.8 "Text tool" — create, edit, basic typography
- PDR §3.2 "Node types — text" — text content storage

**Defers:**
- Rich text / mixed styles within a text block — data model supports it, implementation is future
- Font picker with Google Fonts — v1 uses system fonts + Inter
- Text on path — not in PDR MVP scope
- Vertical text — not in PDR MVP scope

## 13. Future Enhancements (Documented, Not Implemented)

### 13.1 Rich Text (Per-Range Styles)

The `spans: Vec<TextSpan>` field enables different font/size/weight/color per character range. The editing overlay would use `contenteditable` with inline `<span>` elements. Typography controls would apply to the current text selection, not the entire node.

### 13.2 Font Picker

A panel with searchable font list, preview, and recently-used. Google Fonts integration for web-safe fonts. Custom font upload for brand fonts.

### 13.3 Auto-Height Text Boxes

A third sizing mode where the width is fixed and the height auto-adjusts to content. Combine with fixed-width wrapping.

### 13.4 Text on Path

Attach text to a vector path. Characters follow the path curvature. Common in logo and badge design.
