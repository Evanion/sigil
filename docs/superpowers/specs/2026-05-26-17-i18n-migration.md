# Spec 17 — i18n Migration + CI Guard + Multi-Locale Drafts

## Overview

Complete the i18n migration: extract every remaining hardcoded user-facing string in `frontend/src` into the existing `@mbarzda/solid-i18next` framework, add `fr/` and `es/` machine-assisted draft locales alongside the canonical `en/`, and add a CI guard (ESLint rule + locale-parity script) that prevents future hardcoded-string regressions.

This spec is implemented in a single PR. The library + scaffolding already exists; the work is migration + locale drafts + enforcement.

**Depends on:** the existing i18n setup (TransProvider in `App.tsx`, `createTestI18n` in `frontend/src/test-utils/i18n.ts`, 4 namespaces under `frontend/src/i18n/locales/en/`).

---

## 1. Architecture

### 1.1 Library + scaffolding

No library change. Continue using `@mbarzda/solid-i18next` (Solid wrapper) over `i18next`. Both are already installed and configured.

`App.tsx` already wraps the shell in `<TransProvider instance={i18nInstance}>`. The instance is created at startup, loads the four namespaces (`common`, `tools`, `panels`, `a11y`), and is shared across the component tree via `useTransContext`.

### 1.2 Locales

Three locales: `en` (canonical), `fr` (French), `es` (Spanish). Each is a directory under `frontend/src/i18n/locales/` containing four JSON files mirroring the EN namespace structure.

```
frontend/src/i18n/locales/
├── en/
│   ├── a11y.json
│   ├── common.json
│   ├── panels.json
│   └── tools.json
├── fr/
│   ├── a11y.json
│   ├── common.json
│   ├── panels.json
│   └── tools.json
└── es/
    ├── a11y.json
    ├── common.json
    ├── panels.json
    └── tools.json
```

i18next's `fallbackLng: "en"` means a missing key in `fr/` or `es/` resolves to the EN value at runtime. The CI locale-parity check (§4.2) blocks merging if key trees diverge.

### 1.3 Language detection

At startup, the i18next instance reads `navigator.language`, takes the 2-letter prefix, accepts the value if it is one of `["en", "fr", "es"]`, otherwise falls back to `en`. The detection logic lives in a small helper at `frontend/src/i18n/detect-language.ts` so it can be unit-tested independently of i18next.

No persistence in this spec — language preference is per-session. A future settings panel will add a switcher + localStorage persistence.

### 1.4 Namespaces

The four namespaces are stable; this spec does not add new ones. Assignment heuristic for migration:

- **`tools:`** — keyboard-shortcut labels, tool names, tool tooltips.
- **`panels:`** — panel headers, field labels, button text, section names, popover content.
- **`a11y:`** — aria-labels, sr-only status messages, accessible names that differ from visible text.
- **`common:`** — generic buttons (Cancel, Save, Apply), error messages, "Untitled" defaults, units.

When a string fits two namespaces, prefer the more specific one (`tools:` > `panels:` > `a11y:` > `common:`).

### 1.5 Key naming

Dotted within a namespace, kebab-case segments. Examples already in use:

```
panels:regions.leftPanel
tools:select.label
a11y:corner-section.lockedHotspot
common:buttons.cancel
```

Keys are stable identifiers. Renaming a key is a breaking change; prefer adding new keys + migrating consumers in the same commit.

### 1.6 Interpolation + pluralization

Standard i18next syntax:

- **Interpolation:** `t("common:selection.summary", { count: 3 })` reads from `"summary": "{{count}} selected"`.
- **Pluralization:** suffix keys with i18next's plural-rules:
  ```json
  "itemsSelected_one": "{{count}} item selected",
  "itemsSelected_other": "{{count}} items selected"
  ```
  French and Spanish drafts fill the appropriate plural forms per CLDR plural rules. Adopt pluralization only when a plural string appears in the inventory — no synthetic plurals.

### 1.7 Machine-translation provenance

Each `fr/` and `es/` JSON file gets a top-level `_meta` block:

```json
{
  "_meta": {
    "source": "machine-assisted",
    "generatedAt": "2026-05-26",
    "needsNativeReview": true
  },
  "regions": { "leftPanel": "Panneau de gauche" }
}
```

Native-speaker review (in a later spec) removes the `_meta` block. The CI locale-parity script (§4.2) skips any top-level key starting with `_`, so divergent metadata between locales does not fail the build.

## 2. Migration scope + completeness

### 2.1 Inventory pass

Before extraction, generate an inventory of every hardcoded user-facing string in `frontend/src` (excluding `**/*.test.*`, `**/*.stories.*`, `**/__tests__/**`). The inventory is produced by the §4.1 ESLint rule running in report-only mode and committed once to `docs/superpowers/specs/2026-05-26-17-i18n-migration-inventory.md` as a snapshot artifact (not source of truth). The artifact is committed at the start of migration for reviewer reference; once migration completes the inventory is stale and may be deleted in the same PR or in a follow-up cleanup.

The inventory enumerates each violation with file:line, the literal string, and a proposed `namespace:key`. The migration commits work through this list.

### 2.2 "Done" criteria

The migration is complete when:

1. Every entry in the inventory has been replaced with a `t("namespace:key")` call.
2. The ESLint rule (§4.1) reports zero violations on the entire `frontend/src` tree.
3. The locale-parity script (§4.2) reports zero key-tree divergences (excluding `_meta`).
4. `fr/` and `es/` locale files contain machine-translated values for every EN key.
5. Per-locale smoke test (§5.2) passes for each of the three languages.
6. Full frontend test suite passes.

### 2.3 Allowlist conventions

Genuinely-not-user-facing or canonical-string literals are allowed via a same-line or adjacent comment:

```tsx
// i18n-allow: node name is user-typed content, not a translatable label
<title>{node.name}</title>
```

The ESLint rule's standard `eslint-disable-next-line i18next/no-literal-string` works; the `// i18n-allow: <rationale>` convention sits adjacent to it so reviewers see the reason at a glance. A small CI grep (§4.3) flags `eslint-disable` lines for `i18next/no-literal-string` that lack a same-line or adjacent `i18n-allow:` rationale.

### 2.4 Migration order

Files are migrated in batches by namespace, smallest to largest:

1. **`tools:` batch** — `frontend/src/tools/` + tool-related panel files.
2. **`a11y:` batch** — accessibility labels across the codebase.
3. **`panels:` batch** — all remaining panel files.
4. **`common:` batch** — generic buttons, fallbacks, units, error messages.

Each batch is one or more commits. The final commit lands the CI guard turned on.

## 3. fr/es draft generation

### 3.1 Source

Machine-assisted drafts produced during this spec's implementation. The translator runs an LLM pass over each `en/*.json` file producing a parallel `fr/*.json` and `es/*.json` with the same key tree.

### 3.2 Quality bar

- Plausible but imperfect. Subject to native-speaker review later.
- Each fr/es JSON file MUST include the `_meta.needsNativeReview: true` flag.
- Plural-rule keys (`_one`, `_other`, plus CLDR-required forms for fr/es) MUST be filled — fr uses `_one` / `_other`; es uses `_one` / `_other`.
- Interpolation placeholders (`{{var}}`) MUST be preserved exactly.

### 3.3 Translator process

Out of scope for this spec. The native-speaker review pass is a future task — when it lands, removing the `_meta` block is the indicator that a locale has been reviewed.

## 4. CI guard

### 4.1 ESLint rule

Use `eslint-plugin-i18next`'s `no-literal-string` rule. Add to `frontend/.eslintrc.cjs` (or wherever the project's ESLint config lives):

```js
"i18next/no-literal-string": ["error", {
  mode: "jsx-text-only",
  "jsx-attributes": {
    "include": [
      "aria-label",
      "aria-description",
      "aria-placeholder",
      "aria-roledescription",
      "aria-valuetext",
      "placeholder",
      "title",
      "alt"
    ]
  },
  "callees": {
    "include": ["setAnnouncement", "toast", "showToast"]
  },
  "words": {
    "exclude": [
      "^[a-z-]+$",
      "^https?://",
      "^#[0-9a-fA-F]{3,8}$",
      "^[0-9.]+(px|rem|em|%)$"
    ]
  }
}]
```

The rule runs as part of the existing `pnpm lint` step in CI.

### 4.2 Locale-parity script

A small Node script at `frontend/scripts/check-locale-parity.mjs` verifies that every namespace JSON under `fr/` and `es/` has exactly the same set of keys as the EN counterpart, excluding any top-level key starting with `_`.

The script:

1. Loads each `en/*.json`.
2. For each other locale (`fr`, `es`), loads the same namespace file.
3. Recursively walks both key trees; reports missing or extra keys.
4. Exits non-zero if any divergence found.

Wire into CI as a new job (sibling to `kobalte-import-discipline` and `discriminated-union-sentinels`):

```yaml
locale-parity:
  name: Locale Parity
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@<sha>
    - name: Verify fr/ and es/ key trees match en/
      run: node frontend/scripts/check-locale-parity.mjs
```

Adding a new key to `en/` MUST add it to `fr/` and `es/` in the same commit (machine-translated for fr/es).

### 4.3 Allowlist-rationale grep

A CI step (or extension of an existing one) greps for `eslint-disable.*i18next/no-literal-string` lines that lack a same-line or immediately-preceding `i18n-allow:` rationale comment. Strict enforcement: any unannotated disable fails the build. The intent is that legitimate exceptions are rare and always justified inline; if the count grows beyond a handful, the project should reconsider what the lint rule covers rather than silently accumulate unannotated disables.

## 5. Testing

### 5.1 Existing tests

The `createTestI18n` helper at `frontend/src/test-utils/i18n.ts` is already in use. Any test that renders a translatable component MUST wrap it in `<TransProvider instance={await createTestI18n()}>`. The helper loads all four EN namespaces.

Tests that previously asserted hardcoded English strings (`expect(button.textContent).toBe("Save")`) continue to assert against the same English values — the EN locale file is now the source of truth and i18next renders the EN string by default. No assertion rewrites needed.

### 5.2 Per-locale smoke test

New test at `frontend/src/i18n/__tests__/locale-rendering.test.tsx`:

- For each of `en`, `fr`, `es`, mount a representative panel (e.g., the App shell's left-panel region wrapped in a `TransProvider` with that locale).
- Assert the rendered text matches the locale file's value for one known key per namespace.

Catches catastrophic locale-load failures and namespace-not-loaded bugs.

### 5.3 Parity-script self-test

`frontend/scripts/__tests__/check-locale-parity.test.mjs` verifies the script:

- Returns zero exit code when key trees match.
- Returns non-zero and reports missing keys when `fr/common.json` is missing a key that `en/common.json` has.
- Skips `_meta` top-level keys correctly.

### 5.4 ESLint rule smoke test

`frontend/src/i18n/__tests__/eslint-rule-smoke.test.ts`:

- Programmatically loads the project's ESLint config.
- Lints a sample source string containing both an allowed and a disallowed literal.
- Asserts ESLint reports exactly one violation (the disallowed one).

Catches accidental rule-disable in `eslintrc`.

## 6. Input Validation

No new data types or deserialization boundaries introduced. The i18n library handles JSON parsing of locale files at startup; any malformed JSON would fail import. The CI parity script's JSON load wraps in try/catch and reports the failing file.

No user-facing parameter changes. Language detection accepts only the closed set `{en, fr, es}` and falls back to `en` for any other input — no injection vector.

## 7. PDR Traceability

The PDR (`docs/superpowers/specs/2026-04-01-agent-designer-design.md`) lists i18n as MVP infrastructure under "Internationalization." This spec satisfies that line item by:

- Selecting and verifying the framework (`@mbarzda/solid-i18next`).
- Establishing three locales (en/fr/es) with sufficient draft content to demonstrate end-to-end multi-locale rendering.
- Locking the gate against future regressions via the ESLint rule + locale-parity script.

No PDR features are explicitly deferred. The settings-panel language switcher is mentioned in the PDR but is a separate UX surface — this spec leaves the framework ready for it.

## 8. Consistency Guarantees

### 8.1 Atomicity

- Adding a new translatable key MUST add the EN value AND the fr/es draft values in the same commit. The parity script enforces this.
- Migration commits MUST leave the test suite passing — partial migrations of a single file are acceptable across commits, but every commit produces a working app.

### 8.2 Invariants

- For every key `K` under any namespace, `en[K]`, `fr[K]`, `es[K]` all exist (or all three do not exist).
- The `_meta` block exists only in `fr/*.json` and `es/*.json` (never in `en/*.json`).
- The ESLint rule produces zero violations on every commit AFTER the migration's final commit.

### 8.3 Partial failure

- If the parity script fails on a PR, the PR is blocked at CI Gate. No partial merges.
- If a runtime locale-load failure occurs (file missing, JSON malformed), i18next's `fallbackLng: "en"` keeps the app functional. The per-locale smoke test catches this in CI.

## 9. Deferred / out of scope

- Native-speaker review of fr/es drafts. Tracked as a follow-up task; the `_meta.needsNativeReview` flag is the visible reminder.
- Language switcher UI. Deferred to the future settings/preferences panel spec.
- Locale persistence (localStorage / user profile). Deferred to the same future spec.
- Additional locales beyond en/fr/es. Adding more is a small commit once the framework is locked.
- RTL languages (Arabic, Hebrew). Requires CSS logical-properties audit; not needed for en/fr/es.
- Date/time/number formatting via i18next. Not needed by any current string; defer.
- Translator workflow tooling (translation memory, glossary, etc.). Out of scope; the project is small enough that JSON-edit-in-PR is fine.

## 10. Performance Considerations

- Bundle size: i18next + the wrapper add a fixed ~50 KB (gzip) that's already in the bundle. Locale JSON files are ~3 KB each (gzip), three of them load at startup ~9 KB. Acceptable.
- Startup cost: i18next initialization is synchronous after the JSON loads. Negligible.
- Runtime cost: `t()` is a hash lookup; sub-microsecond per call. Negligible.
- Locale switching at runtime (when the future switcher lands) reloads the active locale's namespaces and re-renders subscribers. Acceptable for an explicit user action.
