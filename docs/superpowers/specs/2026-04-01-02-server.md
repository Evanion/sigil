# Spec 02: Server

> Sub-spec of the [Agent Designer PDR](2026-04-01-agent-designer-design.md)

## Revision History

- **v2 (2026-04-02):** Migrated from REST + raw WebSocket to unified GraphQL API over WebSocket transport. Added `async-graphql`. Removed REST endpoints and manual WebSocket protocol.
- **v1 (2026-04-01):** Initial spec — Axum HTTP server with REST endpoints + raw WebSocket for state sync.

## Overview

The application shell — an Axum-based server that serves the frontend SPA, exposes a GraphQL API over a persistent WebSocket connection for real-time state sync, handles file I/O, and discovers/resolves the workfile hierarchy.

## Responsibilities

- **Serve frontend SPA** — static file serving of the built frontend assets
- **GraphQL API** — unified query, mutation, and subscription interface over WebSocket (`graphql-ws` protocol) using `async-graphql`
- **File I/O** — read/write workfiles (`.sigil/` directories) to disk using atomic write-then-rename
- **Workfile discovery** — scan workspace directories, find all `.sigil/` directories, resolve inheritance hierarchy by walking parent directories
- **File watching** — detect external changes to `.sigil/` directories (e.g., git pull, IDE edits) and rebuild token/component contexts
- **Asset management** — store and serve images, fonts from workfile asset directories
- **UUID generation** — generate UUIDs for new nodes, components, tokens (core crate never generates UUIDs)
- **Port configuration** — configurable via `PORT` env var, default 4680

## GraphQL Architecture

All client communication flows through a single persistent WebSocket connection using the `graphql-ws` protocol. No separate REST endpoints or raw WebSocket protocol.

### Transport

- **Endpoint:** `/graphql` — accepts both HTTP POST (for tools/debugging) and WebSocket upgrade (for the app)
- **Protocol:** `graphql-ws` (the modern WebSocket sub-protocol for GraphQL, supported by `async-graphql`)
- **Single connection:** The frontend establishes one WebSocket connection on load. Queries, mutations, and subscriptions all flow through it.

### Schema Overview

```graphql
type Query {
  # Document state
  document: DocumentInfo!
  documentFull: FullDocument!
  page(id: ID!): Page
  node(uuid: ID!): Node

  # Collections
  pages: [Page!]!
  tokens: [Token!]!
  components: [ComponentDef!]!

  # Workfiles
  workfiles: [Workfile!]!
}

type Mutation {
  # Node operations
  createNode(input: CreateNodeInput!): CreateNodeResult!
  deleteNode(uuid: ID!): DeleteNodeResult!
  renameNode(uuid: ID!, newName: String!): Node!
  setTransform(uuid: ID!, transform: TransformInput!): Node!
  setVisible(uuid: ID!, visible: Boolean!): Node!
  setLocked(uuid: ID!, locked: Boolean!): Node!
  setTextContent(uuid: ID!, content: String!): Node!

  # Style operations
  setFills(uuid: ID!, fills: [FillInput!]!): Node!
  setStrokes(uuid: ID!, strokes: [StrokeInput!]!): Node!
  setOpacity(uuid: ID!, opacity: Float!): Node!
  setBlendMode(uuid: ID!, blendMode: BlendMode!): Node!
  setEffects(uuid: ID!, effects: [EffectInput!]!): Node!
  setConstraints(uuid: ID!, constraints: ConstraintsInput!): Node!

  # Tree operations
  reparentNode(uuid: ID!, newParentUuid: ID!, position: Int!): Node!
  reorderChildren(uuid: ID!, newPosition: Int!): Node!

  # Transition operations
  addTransition(input: TransitionInput!): Transition!
  removeTransition(id: ID!): Boolean!
  updateTransition(id: ID!, input: TransitionInput!): Transition!

  # Token operations
  addToken(input: TokenInput!): Token!
  removeToken(name: String!): Boolean!
  updateToken(input: TokenInput!): Token!

  # Component operations
  addComponent(input: ComponentDefInput!): ComponentDef!
  removeComponent(id: ID!): Boolean!

  # History
  undo: UndoRedoResult!
  redo: UndoRedoResult!
}

type Subscription {
  # Fine-grained node change subscriptions
  nodeCreated: NodeCreatedEvent!
  nodeUpdated: NodeUpdatedEvent!
  nodeDeleted: NodeDeletedEvent!

  # Document-level change stream
  documentChanged: DocumentChangedEvent!
}
```

### Key Design Decisions

**UUID-based addressing:** All mutations accept node UUIDs, not arena-local NodeIds. The server resolves UUID → NodeId internally via `arena.id_by_uuid()`. Arena indices never cross the API boundary.

**Mutations capture old state server-side:** The client sends only the new value (e.g., `setTransform(uuid, newTransform)`). The server reads the current state before applying the mutation, constructs the full `SerializableCommand` with both old and new values, and executes through `Document::execute`. This means the client doesn't need to track old state for undo.

**Subscriptions deliver deltas:** The `documentChanged` subscription sends the specific mutation that occurred (as a discriminated union), not a full document snapshot. Clients apply deltas locally. Full state is fetched only on initial connection and reconnect.

**Originator exclusion:** Subscription events include a `clientId` field. The originating client filters out its own events (same pattern as the current broadcast).

### Server-Side Implementation

```
GraphQL mutation arrives (e.g., createNode)
  → resolve UUID to NodeId if needed
  → read old state for undo
  → construct Command struct
  → call Document::execute(cmd)
  → handle SideEffects (file I/O)
  → publish to subscription stream
  → return result to caller
  → signal debounced persistence
```

`async-graphql` integrates with Axum via `async-graphql-axum`. The GraphQL schema object holds a reference to the shared `AppState` (document, broadcast channel, etc.).

## Workspace Mounting

Users mount their project directories into the container. The server scans for `.sigil/` directories within the configured workspace root.

### Configuration

| Env Var | Default | Description |
|---|---|---|
| `WORKSPACE_DIR` | `/workspaces` | Root directory to scan for `.sigil/` workfiles |
| `PORT` | `4680` | Server port |
| `STATIC_DIR` | `/usr/local/share/sigil/frontend` | Frontend SPA assets directory |

### Single Project

```bash
docker run -p 4680:4680 \
  -v ~/my-project:/workspaces/my-project \
  ghcr.io/evanion/sigil:dev
```

### Multiple Projects

```bash
docker run -p 4680:4680 \
  -v ~/hobby-project:/workspaces/hobby \
  -v ~/work-project:/workspaces/work \
  ghcr.io/evanion/sigil:dev
```

### Monorepo with Multiple Workfiles

```
/workspaces/monorepo/
├── design.sigil/              # Root — shared tokens, base components
├── apps/
│   ├── web/
│   │   └── client.sigil/      # Inherits from design.sigil
│   └── mobile/
│       └── mobile.sigil/      # Inherits from design.sigil
```

The inheritance hierarchy is resolved by walking up from each `.sigil/` directory toward the `WORKSPACE_DIR` root.

## Workfile Discovery

### Scan Process

On startup and when file changes are detected:

1. Recursively scan `WORKSPACE_DIR` for directories matching `*.sigil/`
2. For each discovered workfile, walk up parent directories to find ancestor `.sigil/` directories
3. Build the inheritance tree (closer ancestors override farther ones)
4. Load and resolve token contexts through the hierarchy
5. Make all workfiles available via the GraphQL API

### File Watching

The server uses `notify` crate to detect external changes:

1. File watcher detects a change within a `.sigil/` directory
2. Server checks expected-write set (prevent feedback loops from own saves)
3. If external: re-read affected workfiles, rebuild resolved contexts
4. Publish update via GraphQL subscription

## State Sync Model

Human and agent edits flow through the same operation pipeline:

1. Frontend sends GraphQL mutations over WebSocket
2. MCP tools invoke operations directly on core (same process)
3. Server applies through core engine via `Document::execute`
4. Server publishes subscription events to all connected clients
5. Server persists changes to disk (debounced)

### Network Security

- CORS uses explicit origin allowlist (never `*`); development mode gated behind env var
- WebSocket upgrade validates Origin header
- Maximum message size enforced (named constant)
- Graceful shutdown: SIGTERM/SIGINT → stop accepting connections → drain → bounded timeout

### File Persistence Safety

- All file writes use atomic temp-then-rename pattern
- Manifest validated against actual files on disk before loading
- File names use UUIDs (collision-free)
- Flush dirty documents on graceful shutdown
