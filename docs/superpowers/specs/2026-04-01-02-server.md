# Spec 02: Server

> Sub-spec of the [Agent Designer PDR](2026-04-01-agent-designer-design.md)

## Overview

The application shell — an Axum-based HTTP server that serves the frontend SPA, manages WebSocket connections for real-time state sync, handles file I/O, and discovers/resolves the workfile hierarchy.

## Responsibilities

- **Serve frontend SPA** — static file serving of the built frontend assets
- **WebSocket state sync** — accept connections from the frontend, receive intent-based operations, apply them through the core engine, broadcast updated state
- **File I/O** — read/write workfiles (`.sigil/` directories) to disk
- **Workfile discovery** — scan workspace directories, find all `.sigil/` directories, resolve inheritance hierarchy by walking parent directories
- **File watching** — detect external changes to `.sigil/` directories (e.g., git pull, IDE edits) and rebuild token/component contexts
- **Asset management** — store and serve images, fonts from workfile asset directories
- **UUID generation** — generate UUIDs for new nodes, components, tokens (core crate never generates UUIDs)
- **Port configuration** — configurable via `PORT` env var or `--port` CLI flag, default 4680

## Workspace Mounting

Users mount their project directories into the container. The server scans for `.sigil/` directories within the configured workspace root.

### Configuration

| Env Var | Default | Description |
|---|---|---|
| `WORKSPACE_DIR` | `/workspaces` | Root directory to scan for `.sigil/` workfiles |
| `PORT` | `4680` | HTTP/WebSocket server port |
| `STATIC_DIR` | `/usr/local/share/sigil/frontend` | Frontend SPA assets directory |

### Single Project

Mount a single repo into a subdirectory of `/workspaces`:

```bash
docker run -p 4680:4680 \
  -v ~/my-project:/workspaces/my-project \
  ghcr.io/evanion/sigil:dev
```

Server scans `/workspaces/my-project/` and finds all `.sigil/` directories within it.

### Multiple Projects

Mount multiple repos as siblings under `/workspaces`:

```bash
docker run -p 4680:4680 \
  -v ~/hobby-project:/workspaces/hobby \
  -v ~/work-project:/workspaces/work \
  ghcr.io/evanion/sigil:dev
```

Server scans `/workspaces/` and discovers `.sigil/` directories in both projects. The UI provides a workspace picker to switch between them.

### Monorepo with Multiple Workfiles

A single mount with multiple `.sigil/` directories at different levels:

```
/workspaces/monorepo/
├── design.sigil/              # Root — shared tokens, base components
├── apps/
│   ├── web/
│   │   └── client.sigil/      # Inherits from design.sigil
│   └── mobile/
│       └── mobile.sigil/      # Inherits from design.sigil
```

The inheritance hierarchy is resolved by walking up from each `.sigil/` directory toward the `WORKSPACE_DIR` root. `client.sigil` inherits from `design.sigil` because it's an ancestor directory.

### Custom Mount Point

Override the workspace root for non-standard layouts:

```bash
docker run -p 4680:4680 \
  -v ~/projects:/data \
  -e WORKSPACE_DIR=/data \
  ghcr.io/evanion/sigil:dev
```

### Dev Container Integration

In a dev container stack, the workspace is typically already mounted:

```json
{
  "mounts": [
    "source=${localWorkspaceFolder},target=/workspaces/project,type=bind"
  ]
}
```

Sigil can run as a service in the same stack, sharing the mount.

## Workfile Discovery

### Scan Process

On startup and when file changes are detected:

1. Recursively scan `WORKSPACE_DIR` for directories matching `*.sigil/`
2. For each discovered workfile, walk up parent directories to find ancestor `.sigil/` directories
3. Build the inheritance tree (closer ancestors override farther ones)
4. Load and resolve token contexts through the hierarchy
5. Make all workfiles available via the API and UI

### File Watching

The server uses a file watcher (`notify` crate) to detect external changes to `.sigil/` directories:

1. File watcher detects a change (new/modified/deleted file within a `.sigil/` directory)
2. Server checks if the change matches an expected write (from its own save operations):
   - If yes → ignore (prevent feedback loops)
   - If no → treat as external change
3. Server re-reads the affected workfile(s)
4. Server rebuilds the resolved `TokenContext` and component definitions from the updated hierarchy
5. Server pushes the updated context into the core engine
6. Server broadcasts the change to connected clients (frontend + MCP agents)

**Expected write tracking:** The server maintains a set of `(file_path, content_hash)` for writes it has initiated. When the file watcher fires, it checks this set. Matches are removed from the set and ignored. Non-matches are treated as external changes.

## State Sync Model

Human and agent edits flow through the same operation pipeline:
1. Frontend sends operations via WebSocket; MCP tools invoke operations directly on core
2. Server applies operations through the core engine
3. Server calls `document.resolve_layout()` to compute any dirty layouts
4. Server broadcasts updated state to all connected clients
5. Server persists changes to disk (debounced to avoid excessive writes)

### Broadcast Format

The server broadcasts `BroadcastCommand` messages (forward-only, no undo state) to connected clients. Each client maintains its own undo history locally.
