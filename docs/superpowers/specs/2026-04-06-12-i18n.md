# Spec 12 — Internationalization (i18n)

## Overview

Sets up internationalization infrastructure before hardcoded strings multiply further. Uses `solid-i18next` (i18next wrapper for Solid.js) for translation strings with ICU message format, and Kobalte's `I18nProvider` for locale + text direction. English is the only shipped locale in v1 — the infrastructure enables community translations.

**Depends on:** Spec 07 (component library — Kobalte integration), Spec 08 (shell)

---

## 1. Architecture

### 1.1 Two Layers

**Kobalte I18nProvider** — wraps the app root. Provides `locale` and `direction` signals to all Kobalte components (NumberField locale formatting, DatePicker, etc.). Reads the user's browser locale by default, overridable via settings.

**solid-i18next** — provides the `t()` translation function as a Solid-reactive accessor. Loads JSON namespace files. Supports ICU message format (plurals, interpolation, select, number/date formatting).

Both layers share the same locale signal — when the user switches locale, both update.

### 1.2 Why i18next

- ICU message format handles pluralization rules per locale (English: 1/other, Arabic: zero/one/two/few/many/other, etc.)
- Namespace support for code-splitting translations by feature area
- Interpolation with type safety via `t('key', { count: 5 })`
- Fallback chains (e.g., `fr-CA` → `fr` → `en`)
- Battle-tested with 50M+ weekly downloads
- `solid-i18next` provides reactive `useTransContext()` hook

## 2. File Structure

```
frontend/src/i18n/
  index.ts                    — i18next initialization + Solid integration
  types.ts                    — typed namespace keys (generated or manual)

frontend/src/i18n/locales/
  en/
    common.json               — shared strings (ok, cancel, save, delete, etc.)
    tools.json                — toolbar and tool names
    panels.json               — panel labels, section headers, field labels
    a11y.json                 — aria-labels, screen reader announcements
    effects.json              — effect type names, field labels
    color.json                — color picker labels, space names
```

### 2.1 Namespace Strategy

Translations are split by feature area to enable lazy loading and reduce bundle size:

| Namespace | Content | Example keys |
|-----------|---------|-------------|
| `common` | Shared UI strings | `ok`, `cancel`, `delete`, `undo`, `redo` |
| `tools` | Tool names + tooltips | `tool.select`, `tool.frame`, `tool.rectangle` |
| `panels` | Panel/section labels | `panel.design`, `panel.layers`, `section.fill` |
| `a11y` | Accessibility strings | `aria.colorPicker`, `aria.addFill`, `announce.fillAdded` |
| `effects` | Effects panel | `effect.dropShadow`, `effect.blur`, `field.offsetX` |
| `color` | Color picker | `space.srgb`, `space.oklch`, `field.hex` |

### 2.2 Key Naming Convention

Dot-separated, hierarchical: `namespace:section.element.state`

Examples:
- `panels:fill.empty` → "No fills"
- `panels:stroke.add` → "Add stroke"
- `a11y:fill.removed` → "Fill removed. {{count}} fills total."
- `common:action.delete` → "Delete"

## 3. Initialization

```typescript
// frontend/src/i18n/index.ts
import i18next from "i18next";
import { createI18nContext, I18nContext } from "solid-i18next";
import commonEn from "./locales/en/common.json";
import toolsEn from "./locales/en/tools.json";
// ... other namespaces

i18next.init({
  lng: navigator.language,          // browser default
  fallbackLng: "en",
  defaultNS: "common",
  interpolation: { escapeValue: false }, // Solid handles XSS
  resources: {
    en: {
      common: commonEn,
      tools: toolsEn,
      // ...
    },
  },
});
```

### 3.1 App Root Wiring

```tsx
// App.tsx
import { I18nProvider } from "@kobalte/core/i18n";
import { I18nContext, createI18nContext } from "solid-i18next";

<I18nContext.Provider value={createI18nContext()}>
  <I18nProvider locale={currentLocale()}>
    <Shell />
  </I18nProvider>
</I18nContext.Provider>
```

## 4. Usage Pattern

### 4.1 In Components

```tsx
import { useTransContext } from "solid-i18next";

function AppearancePanel() {
  const [t] = useTransContext();

  return (
    <button aria-label={t("panels:fill.add")}>+</button>
    <p>{t("panels:fill.empty")}</p>
  );
}
```

### 4.2 Pluralization (ICU)

```json
{
  "fill.added": "Fill added. {{count}} fill total.",
  "fill.added_plural": "Fill added. {{count}} fills total."
}
```

```tsx
announce(t("a11y:fill.added", { count: newFills.length }));
```

### 4.3 aria-labels

All `aria-label` attributes use `t()` — these are the most critical strings for i18n since screen reader users depend on them being in their language.

```tsx
<button aria-label={t("a11y:fill.remove")}>×</button>
```

## 5. Migration Plan

### 5.1 Extraction Priority

Migrate existing hardcoded strings in this order:

1. **aria-labels and screen reader announcements** — accessibility-critical
2. **Panel headers and section titles** — visible, high-frequency
3. **Button labels and tooltips** — visible, medium-frequency
4. **Empty states and placeholder text** — visible, low-frequency
5. **Error messages** — rare but important

### 5.2 Migration Approach

No big-bang rewrite. Each file is migrated individually:
1. Import `useTransContext`
2. Replace hardcoded strings with `t()` calls
3. Add corresponding keys to the English locale JSON
4. Run tests (strings change from literal to function calls but behavior is identical)

The migration can be done file-by-file across multiple PRs.

---

## 6. Locale Persistence

User's locale preference is persisted to `localStorage` under `sigil-locale`. On app start:
1. Read `sigil-locale` from localStorage
2. If present, use it. If not, use `navigator.language`
3. Initialize i18next with the resolved locale
4. Pass to Kobalte's `I18nProvider`

A locale switcher in settings (future) writes to `sigil-locale` and calls `i18next.changeLanguage()`. Not in scope for v1 — English only, but the infrastructure is ready.

---

## 7. Input Validation

- **Locale string:** BCP47 format, validated by i18next (falls back to `en` on invalid).
- **Translation keys:** Typed namespace keys prevent typos at compile time (optional, can add later with `i18next-parser` or manual types).
- **Interpolation values:** i18next escapes by default; disabled because Solid handles XSS via JSX.

---

## 8. Consistency Guarantees

- All strings go through `t()` — no hardcoded user-facing text in JSX.
- Namespace files are the single source of truth for each feature area's strings.
- Fallback chain ensures missing translations show English rather than raw keys.
