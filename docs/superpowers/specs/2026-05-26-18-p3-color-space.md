# Spec 18 — Display-P3 Color Space Support

**Status:** Draft
**Author:** Mikael Pettersson (brainstormed 2026-05-26)
**Depends on:** Spec 09 (color picker), Spec 13 (token references — colors are token-valued)
**Estimated PR size:** 600-900 LOC including tests + locale strings

---

## Overview

Sigil's data model already includes `Color::DisplayP3 { r, g, b, a }` as a serde variant (see `crates/core/src/node.rs:516`). The frontend and Rust math today treat P3 channels as sRGB approximation — the P3 option is hidden from the color picker, and `colorToCss` in `frontend/src/panels/token-helpers.ts` renders any non-sRGB color as gray (`rgba(128, 128, 128, 1)`). This spec lights up proper Display-P3 support end-to-end through the editor: matrix-correct conversions, picker emits P3-tagged colors when the user picks them, and the canvas + previews render those colors wide-gamut on capable displays.

Out of scope: CSS token export (`sigil-cli` binding output) emitting `color(display-p3 …)` is a separate spec (see "Deferred" below).

---

## Motivation

- The Rust core already accepts and stores `Color::DisplayP3` from MCP / GraphQL but the picker can't emit it, so the variant is reachable only via direct file edits.
- Wide-gamut displays (modern iOS / macOS / many Windows panels) are widespread; designers expect to be able to specify vibrant brand colors that exceed sRGB.
- The existing "treated as sRGB approximation" code path is a known footgun — copy-pasting a P3 color from another tool stores faithful P3 channels that then render as gray on Sigil's canvas. Faithful storage with no rendering pipeline is worse than no P3 support at all.

---

## Goals

1. **Math correctness.** `colorToSrgb` and `srgbToColor` apply the published P3↔sRGB matrices via XYZ (D65) with proper sRGB transfer encoding on both legs. Round-trip identity within `1e-6` for in-gamut colors. Out-of-gamut signal preserved (no clipping at conversion).
2. **Picker support.** The color picker exposes a fourth display mode `display_p3` alongside `srgb`, `oklch`, `hsl`. Picking it changes both the field labels/ranges and the storage tag emitted by the next commit. Switching modes preserves the visual color (channels convert via XYZ).
3. **Wide-gamut canvas rendering.** P3-tagged colors render through `color(display-p3 r g b / a)` CSS notation. The main editor canvas and the picker canvases (`AlphaStrip`, `HueStrip`, `ColorArea`) acquire 2D contexts with `colorSpace: "display-p3"` so the values render wide-gamut on P3 displays.
4. **Out-of-gamut warning.** The existing hex-input warning glyph extends to fire for P3-tagged colors that fall outside sRGB; designers see a visible cue that their color won't reproduce on legacy displays.
5. **Parity between Rust and TypeScript.** Both stacks share a single set of matrix constants (cited to W3C `css-color-4` §10) and parity-test against shared input/output vectors in `tests/fixtures/parity/p3-color-conversions.json`.

---

## Non-goals

- Implementing a third or fourth color profile (Rec.2020, Adobe RGB, ProPhoto). The data model could carry them as future enum variants; this spec is P3 only.
- Migrating existing sRGB-stored colors. Existing documents stay sRGB-tagged; users opt into P3 by choosing the picker mode.
- Implementing image-data conversions for `<canvas getImageData>`. Reading pixels back from a P3 canvas to inspect channels is a separate concern; Sigil doesn't do this today.
- CSS export of P3 tokens via `sigil-cli` / `@sigil/css` (deferred — see Open questions).
- ICC color profile awareness for imported assets (deferred indefinitely; we trust the file format's color space tag at the data-model level).

## §1 Storage model

No schema change. `Color::DisplayP3 { r, g, b, a }` already exists in `crates/core/src/node.rs:516` and round-trips via `#[derive(Deserialize)]`. Channels are 4 × `f64`; the API-level semantic range is `[0.0, 1.0]` in the Display-P3 gamut. Deserialization tolerates wider values to allow round-tripping of OkLCH wide-gamut conversions — see §7 for the detailed validation table.

### Mode is the storage tag

When the picker's `ColorDisplayMode` is `"display_p3"`, the next commit emits `Color::DisplayP3`. Mode is the user's expressed intent; we honor it even if the picked color is in-gamut for sRGB. This matches Figma's behavior and preserves round-trip identity (a designer who picks `#FF0000` in P3 mode reads back P3 red, not sRGB red).

### Cross-mode display

When the picker mode changes (e.g., sRGB → P3), the current visual color is preserved: channels convert via XYZ; new channels replace the displayed ones; storage tag changes on next commit. No "are you sure?" prompt — the conversion is informationally lossless (within sRGB gamut) and the OOG warning surfaces lossy transitions.

When a P3-tagged color is later viewed with the picker in sRGB mode, the picker shows the sRGB-clipped channels in numeric fields and the OOG warning fires. Storage tag is preserved until the user explicitly commits a new color in the new mode.

## §2 Frontend math (`color-math.ts`)

Three function bodies change:

- `colorToSrgb(color)` for `display_p3`: apply P3→XYZ→sRGB matrix sequence. Output channels are unclamped (may be `< 0` or `> 1` — out-of-sRGB-gamut signal).
- `srgbToColor(r, g, b, alpha, "display_p3")`: apply sRGB→XYZ→P3 matrix sequence. Output channels unclamped (so wide-gamut sRGB inputs, if they ever happen, round-trip).
- `isOutOfSrgbGamut(color)` for `display_p3`: matrix-convert to sRGB, check any channel falls outside `[0, 1]` to within an epsilon (`1e-7`).

### Matrix constants

A new module `frontend/src/components/color-picker/color-matrices.ts` exports:

```ts
// All matrices are CIE XYZ (D65), per W3C css-color-4 §10.
// https://www.w3.org/TR/css-color-4/#color-conversion-code

export const SRGB_TO_XYZ_D65: ReadonlyArray<ReadonlyArray<number>>;
export const XYZ_TO_SRGB_D65: ReadonlyArray<ReadonlyArray<number>>;
export const DISPLAY_P3_TO_XYZ_D65: ReadonlyArray<ReadonlyArray<number>>;
export const XYZ_TO_DISPLAY_P3_D65: ReadonlyArray<ReadonlyArray<number>>;

// sRGB transfer function and its inverse (Display-P3 uses the same pair).
export function srgbEotf(channel: number): number; // gamma-decode: γ → linear
export function srgbOetf(channel: number): number; // gamma-encode: linear → γ
```

`color-math.ts` imports these and composes them — keeping matrix math in a single source of truth (per CLAUDE.md §5 "Validation constants ... MUST be defined in a single source-of-truth module").

## §3 Picker display mode

### Type extension

`frontend/src/components/color-picker/types.ts`:

```ts
export type ColorDisplayMode = "srgb" | "display_p3" | "oklch" | "hsl";
```

Comment in the file (currently noting P3 was removed) is updated to describe the new semantics.

### Switcher

`ColorSpaceSwitcher.tsx` un-comments the P3 option (4 buttons total). The existing roving-tabindex / radio-group keyboard semantics extend naturally. New locale keys:

- `panels:colorPicker.p3Label` = `"P3"` (button label)
- `panels:colorPicker.p3Title` = `"Wide-gamut Display-P3 colors"` (button title attribute)
- `panels:colorPicker.p3HexHint` = `"Hex channels interpreted as Display-P3"` (the P3 badge title near the hex field)

Added to all three locales (en/fr/es) with machine-translated drafts.

### Numeric ranges

In P3 mode, the R/G/B `ValueInput` fields show 0-1 floats with 4 decimal places (matches CSS `color()` syntax and Figma's P3 panel). Step: `0.01` for arrow scrub, `0.1` for shift-arrow.

### Hex input

`HexInput` continues to accept `#rrggbb`. When the picker mode is `display_p3`:
- The decoded `0xRR / 0xGG / 0xBB` values map to `Color::DisplayP3` channels directly (i.e., `#FF0000` ↔ P3 red).
- A small `P3` badge appears immediately after the hex input (similar visual weight to the OOG warning glyph). The badge uses `aria-hidden="true"` (state metadata) and a `title={t("panels:colorPicker.p3HexHint")}` for hover. The mode is also conveyed via the radio button's `aria-checked` state; the badge is reinforcement, not the SR-authoritative label.

The OOG warning behavior is described in §5.

## §4 Display / CSS rendering

### `colorToCss` (`frontend/src/panels/token-helpers.ts`)

Replaced from "gray fallback for any non-sRGB" to per-variant:

```ts
export function colorToCss(color: Color): string {
  if (color.space === "srgb") {
    // existing behavior: rgba(r255, g255, b255, a)
  }
  if (color.space === "display_p3") {
    // color(display-p3 r4 g4 b4 / a) — 4 decimal places, Number.isFinite guards
  }
  if (color.space === "oklch") {
    // existing behavior: convert via colorToSrgb and emit rgba(...) — OkLCH
    // wide-gamut storage stays out of CSS for now (a future spec may emit
    // `oklch(...)` CSS directly).
  }
  // oklab same fallback
}
```

Number.isFinite guards on every interpolated channel per CLAUDE.md §11.

### Canvas color space

Main editor canvas (`frontend/src/canvas/Canvas.tsx` — or wherever the 2D context is acquired) and picker canvases (`AlphaStrip`, `HueStrip`, `ColorArea`) update their `getContext("2d", { colorSpace: "display-p3" })` call. A small wrapper helper `acquireWideGamut2D(canvas)` lives in `frontend/src/canvas/canvas-context.ts` and centralizes the call with the correct options shape; consumers import it.

Gradient interpolation: with the canvas in `display-p3` colorspace, `ctx.createLinearGradient` / `createRadialGradient` interpolates between stops in the canvas's color space (per Canvas2D spec). No explicit configuration needed.

### Browser support

Target: latest 2 versions of Chrome (any Chromium), Firefox, Safari — all support `color(display-p3 ...)` and `<canvas colorSpace="display-p3">`. On a non-P3-capable display, the browser performs color management to map the wide-gamut values into the display's gamut. On unsupported browsers (out-of-scope), the canvas falls back silently to sRGB rendering of clipped channels.

## §5 Out-of-sRGB-gamut warning

The existing `HexInput` warning glyph (added in PR #66 for OkLCH wide-gamut values; uses `panels:colorPicker.outOfGamutShort` and `outOfGamutLong`) extends to P3:

- In any mode, when `isOutOfSrgbGamut(currentColor) === true`, the ⚠ glyph appears next to the hex input.
- Title / aria-label keep the existing strings (no new locale keys needed for the warning text — it already says "Color is outside the sRGB gamut", which is accurate for P3 OOG colors too).
- No "click to clamp" affordance in v1 — designers who want sRGB explicit switch mode.

## §6 Rust core changes + WASM compatibility

### Code changes

**`crates/core/src/tokens/color_convert.rs`:**
- `color_to_srgb(&Color)` — `DisplayP3` arm changes from channel passthrough to proper matrix conversion. Output may have channels outside `[0, 1]` (OOG signal). The existing test `test_color_to_srgb_extracts_display_p3_channels` (line 358) is replaced by `test_color_to_srgb_display_p3_red_matches_w3c_reference` with proper expected values.
- New: `srgb_to_display_p3(r, g, b) -> [f64; 3]` — inverse path for future conversion needs.
- New: `is_out_of_srgb_gamut(&Color) -> bool` — mirrors the frontend helper.

**New module `crates/core/src/color_matrix.rs`** exporting the same constants the frontend module exports, with W3C citation comments. Both stacks load the same numerical values (single source of truth at the workspace level).

### WASM Compatibility (per CLAUDE.md §10)

- **Dependencies:** No new external dependencies. All math is pure `f64` arithmetic.
- **Trait bounds:** None added. The `Color` enum's existing `Send + Sync + 'static` story is unchanged.
- **Randomness / syscalls:** None.
- **Verdict:** Fully WASM-compatible. The matrix module is `no_std`-friendly (uses only `f64::powf` and arithmetic, both available in core).

### Floating-Point Validation (per CLAUDE.md §11)

Add NaN/inf guards in the deserialization path AND a value-construction validator. The current `Color::DisplayP3` derive-deserialize accepts NaN silently; the new path rejects with a typed error.

**Scope question:** the existing `Color::Srgb` variant has the same NaN/inf gap and the same fix shape. Fix both in this PR (single test fixture covers both variants). This satisfies the existing CLAUDE.md rule "No Derive Deserialize on Validated Types" — we move both variants to a manual deserialize that routes through a validating constructor.

## §7 Input Validation Inventory (per CLAUDE.md §10)

| Field | Type | Validation |
|---|---|---|
| `Color::Srgb { r, g, b, a }` | `f64` × 4 | NaN/inf rejected. Channels accepted in `[0.0, 1.0]` for API/MCP input; deserialization tolerates `[−ε, 1+ε]` to allow round-tripping wide-gamut OkLCH→sRGB conversions. |
| `Color::DisplayP3 { r, g, b, a }` | `f64` × 4 | NaN/inf rejected. Channels in `[0.0, 1.0]` for API input. Deserialization accepts any finite value (OOG P3 colors via OkLCH conversion may legitimately exceed `1.0`). |
| `Color::Oklch { l, c, h, a }` | `f64` × 4 | unchanged |
| `Color::Oklab { l, a, b, alpha }` | `f64` × 4 | unchanged |
| `ColorDisplayMode` string | `srgb \| display_p3 \| oklch \| hsl` | exact-match enum |

**New constants in `validate.rs`:** `MIN_COLOR_CHANNEL = 0.0`, `MAX_COLOR_CHANNEL = 1.0`. Used by API-boundary validation; deserialization allows wider tolerance per the table above.

**Frontend mirror:** `frontend/src/store/style-value-validate.ts` (or equivalent) imports the same numeric constants via TypeScript so symmetric validation holds across all transports (per CLAUDE.md §11 "Validation Must Be Symmetric Across All Transports").

## §9 Consistency Guarantees (per CLAUDE.md §10)

- **Atomicity:** a single color commit (any picker change) is atomic at the store level (existing `setFillColor` mutation contract preserved).
- **Cross-field invariants:** the `space` discriminant and the active channel set are validated together — a `Color::DisplayP3` with NaN in `b` is rejected as a single error, not silently coerced.
- **No batch operations introduced.**
- **History:** picker commits coalesce within a drag gesture (existing pattern; unchanged by this spec).

---

## §10 Recursion Safety (per CLAUDE.md §10)

**No recursive data structures or algorithms introduced.** The matrix-multiply helpers are fixed-iteration loops over `3 × 3` matrices; no traversal of user-supplied trees.

## Open questions / deferred

- **CSS / Tailwind export of P3 tokens.** `sigil-cli` and `@sigil/css` currently emit sRGB hex from any color token. P3 tokens deserve `color(display-p3 ...)` output, ideally with a sRGB-hex fallback via `@supports`. Separate spec (Spec 19 candidate).
- **OkLCH-to-P3 auto-promotion.** OkLCH mode in the picker today stores Oklch colors. With P3 in storage, should OkLCH-picked colors outside sRGB but inside P3 be auto-promoted to `Color::DisplayP3`? Decision for v1: NO. OkLCH stays OkLCH. (Two storage paths for wide-gamut, distinct use cases — perceptual vs designer-familiar coordinates.)
- **Canvas image-data round-tripping.** `ctx.getImageData()` on a P3 canvas returns pixels in the canvas's color space (per spec). Sigil doesn't read pixel data today; if a future feature does (e.g., color picker eyedropper from a rendered image), it'll need to handle the colorSpace in `ImageData`. Tracked as a constraint to revisit.
- **OOG-of-P3 colors.** Tokens / files containing `Color::DisplayP3` channels outside `[0, 1]` are theoretically possible (extended-range P3). Validation rejects them at API boundaries. The picker can't produce them. Deserialization tolerates them for round-tripping but the OOG warning surfaces them. No further v1 work needed.
- **Color profile export for raster assets.** Out of scope; we don't handle imported raster assets at the color-managed level today.
